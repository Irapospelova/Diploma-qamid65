- Навигационная панель

- Просмотр информации о приложении, переход по ссылкам.

## 4. Методология и типы тестирования

### Методология

Тестирование будет проводиться с использованием методологии черного ящика, включая ручное и автоматическое UI тестирование.

### Типы тестирования

**Функциональное тестирование**
Будет проверяться:
- Работа всех функций приложения из п.3.

**Юзабилити тестирование**
Будет проверяться:
- Восстановление приложения после сворачивания/разворачивания.
- Работа при оповещениях (звонок, SMS).

**Тестирование безопасности**
Будет проверяться:
- Защита данных пользователей (авторизация).

**Тестирование производительности**
Будет проверяться:
- Скорость загрузки страниц при разных вариантах подключения к сети (wifi, 3G).

**Тестирование совместимости**
Будет проверяться:
- Работа при разной ориентации экрана.
- Совместимость с темами ОС.
- Совместимость с навигационными инструментами телефона (кнопки, жесты)

**Тестирование локализации**
Будет проверяться:
- Перевод элементов интерфейса на язык системы.

**Тестирование установки**
Будет проверяться:
- Верстка страниц приложения и работа функционала на различных устройствах (разные модели телефонов, планшеты).

## 6. Перечень используемых инструментов с обоснованием выбора

### Язык программирования

- JavaScript, Java.

### Системы контроля версий

- GitHub для управления версиями кода и совместной работы.

### Эмулятор/Симулятор

- Android Studio Emulator для тестирования на различных виртуальных устройствах Android.

### Реальные устройства

- OnePlus 9 для ручных тест-кейсов: это устройство обеспечивает тестирование на реальном оборудовании с версией Android 14.

### Библиотеки и инструменты для автоматизации тестирования

- Appium: Инструмент для кроссплатформенного автоматизированного тестирования мобильных приложений. Поддерживает как Android, так и iOS, и позволяет писать тесты на различных языках программирования, включая JavaScript.
- Android Studio.
- wd: Библиотека WebDriver для Node.js, используемая для взаимодействия с Appium Server и мобильными устройствами.
- Mocha: Тестовый фреймворк для JavaScript, используемый для написания и запуска тестов.
- Chai: Библиотека для утверждений в тестах, которая используется вместе с Mocha для проверки результатов тестов.
- Allure: Инструмент для генерации отчетов о тестировании, интегрируемый с Mocha для создания наглядных отчетов о результатах тестов.

### Характеристики реальных устройств

- Android устройства с версией API 29 и выше.

## 7. Создание тест-кейсов на основе чек-листа

Тест-кейсы будут включать шаги для выполнения тестов, ожидаемые результаты.

## 8. Сортировка тест-кейсов по степени автоматизации

### Автоматизируемые тесты

- Аутентификация.
- Просмотр новостей.
- Редактирование новостей.
- Просмотр цитат.
- Переходы по разделам меню.
- Переходы по кнопкам навигационной панели.


### Ручные тесты

- Локализация приложения.
- Работа при оповещениях (звонок, SMS).
- Совместимость с темами ОС.
- Скорость загрузки страниц при разных вариантах подключения к сети (4G, 3G ).
- Работа при разной ориентации экрана.
- Восстановление приложения после сворачивания.
- Совместимость с навигационными кнопками и жестами телефона.

## 9. Выполнение проверок

Тестирование будет проводиться согласно тест-кейсам, с использованием как ручного, так и автоматического тестирования.

## 10. Отчет о тестировании

Отчеты будут включать результаты тестирования положительные и отрицательные. Будут представлены отчеты о выполненных ручных тестах, Allure Reports.

## 11. Временная оценка с учетом рисков

### Оценка времени

- Подготовка тестовой среды: 3 дня.
- Проведение тестирования: 1 день.
- Анализ результатов и отчетность: 2 дня.

### Возможные риски и пути их разрешения

- **Недостаток тестовых данных**:
  - **Риск**: Отсутствие достаточного количества данных для полноценного тестирования.
  - **Путь разрешения**: Подготовка резервных данных заранее, чтобы иметь возможность использовать их при необходимости.

- **Нестабильность тестируемой версии**:
  - **Риск**: Тестируемая версия приложения может содержать критические ошибки или быть нестабильной.
  - **Путь разрешения**: Взаимодействие с разработчиками для быстрой фиксации дефектов.

- **Проблемы с эмуляторами/симуляторами**:
  - **Риск**: Эмуляторы и симуляторы могут работать не так как ожидалось или не полностью эмулировать поведение реальных устройств.
  - **Путь разрешения**: Использование реальных устройств для тестирования. Подготовка реальных устройств заранее.

## 13. Перечень и описание возможных рисков при автоматизации

- **Риск**: Высокая сложность тестов
  - **Решение**: Использование хорошо документированных сценариев и тест-кейсов, которые подробно описывают шаги выполнения и ожидаемые результаты.

- **Риск**: Длительное время разработки тестов
  - **Решение**: Параллельное выполнение ручного тестирования либо частичное замещение некоторых автотестов.

- **Риск**: Проблемы с интеграцией автоматизированных тестов
  - **Решение**: Регулярное обновление и проверка тестов, чтобы убедиться в их корректной работе. Внедрение CI/CD процессов для автоматического выполнения тестов при каждом изменении кода.

- **Риск**: Недостаток тестовых данных
  - **Решение**: Подготовка резервных данных заранее, чтобы иметь возможность использовать их при необходимости.

- **Риск**: Проблемы с эмуляторами/симуляторами
  - **Решение**: Использование реальных устройств для тестирования в случае возникновения проблем с эмуляторами. Подготовка реальных устройств заранее.

## 14. Тестовая документация на проекте

### Документы

- Тест-план. 
  - **Содержимое**: Цели тестирования, объекты тестирования, методология, типы тестирования, планирование времени, управление рисками, перечень специалистов, тестовая документация, критерии начала и окончания тестирования.
- Чек-листы.
  - **Содержимое**: Перечень проверок для различных функций приложения.
- Тест-кейсы.
  - **Содержимое**: Подробное описание шагов для выполнения тестов, ожидаемые результаты.
- Отчеты о ручном тестировании.
  - **Содержимое**: Результаты выполнения тестов, найденные дефекты.
- Отчеты по автотестам (Allure Reports).
  - **Содержимое**: Графическое представление результатов автоматизированных тестов, скриншоты.

## 15. Критерии начала и окончания тестирования

### Критерии начала тестирования

- Доступны все описания функциональности приложения и сценарии использования, которые объясняют, как приложение должно работать и какие функции оно должно выполнять.
- Подготовлены все необходимые тестовые данные (учетная запись пользователя, тестовые новости и цитаты).
- Подготовлена тестовая среда (включая эмуляторы, симуляторы и реальные устройства).

### Критерии окончания тестирования

- Все запланированные тест-кейсы выполнены, результаты зафиксированы.
- Критические и высокоприоритетные дефекты устранены или получены обходные решения.
- В последнем цикле тестирования не обнаружены критические дефекты.
