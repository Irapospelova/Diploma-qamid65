# План тестирования мобильного приложения

## Цели тестирования

Оценить функционирование рабочего приложения, оценить удобство пользования.

## 1. Область применения

Тестирование мобильного приложения, которое рассказывает о хосписе. 

## 2. Объекты тестирования

- Навигационная панель
- Главная страница
- Меню
- Страница авторизации
- Страница Новости
- Страница с цитатами/мыслями
- Страница Информация о приложении

## 3. Реализованный функционал

Приложение включает следующие реализованные функции:

- Кнопки навигационной панели открывают страницы
- Аутентификация пользователей (вход/выход).
- Переход по разделам меню.
- Просмотр, добавление, редактирование новостей.
- Сортировка, фильтрация новостей.
- Просмотр цитат
- Просмотр информации о приложении, переход по ссылкам.

## 4. Методология и типы тестирования

### Методология

Тестирование будет проводиться с использованием методологии черного ящика, включая ручное и автоматическое UI тестирование.

### Типы тестирования

**Функциональное тестирование**

Будет проверяться:
- Работа всех функций приложения из п.3.

**Юзабилити тестирование**

Будет проверяться:
- Логичность и интуитивность навигации по приложению.
- Совместимость с навигационными кнопками в нижней панели телефона

**Тестирование безопасности**

Будет проверяться:
- Защита данных пользователей (авторизация).

**Тестирование производительности**

Будет проверяться:
- Скорость загрузки страниц.

**Тестирование совместимости**

Будет проверяться:
- Работа приложения на различных устройствах (разные модели телефонов, планшеты).
- Работы приложения на версиях ОС Android не ниже 10.

## 8. Сортировка тест-кейсов по степени автоматизации

### Автоматизируемые тесты

- Авторизация пользователей.
- Просмотр новостей.
- Редактирование новостей.
- Просмотр цитат.

## 9. Выполнение проверок

Тестирование будет проводиться согласно тест-кейсам, с использованием как ручного, так и автоматического тестирования.

## 10. Отчет о тестировании

Отчеты будут включать результаты тестирования положительные и отрицательные. Будут представлены отчеты о выполненных ручных тестах, Allure Reports.


## 11. Временная оценка с учетом рисков

### Оценка времени

- Подготовка тестовой среды: 1 день.
- Проведение тестирования: 1 день.
- Анализ результатов и отчетность: 1 день.

### Возможные риски и пути их разрешения

- Недостаток тестовых данных.
Риск: Отсутствие достаточного количества данных для полноценного тестирования.
Путь разрешения: Подготовка резервных данных заранее, чтобы иметь возможность использовать их при необходимости.

- Нестабильность тестируемой версии.
Риск: Тестируемая версия приложения может содержать критические ошибки или быть нестабильной.
Путь разрешения: взаимодействие с разработчиками для быстрой фиксации дефектов.

- Проблемы с эмуляторами/симуляторами.
Риск: Эмуляторы и симуляторы могут работать не так как ожидалось или не полностью эмулировать поведение реальных устройств.
Путь разрешения: Использование реальных устройств для тестирования. Подготовка реальных устройств заранее.

### Команда тестирования

- Девопс инженер (DevOps Engineer): настройка CI/CD

## 13. Перечень и описание возможных рисков при автоматизации

Риск: Высокая сложность тестов
Решение: Использование хорошо документированных сценариев и тест-кейсов, которые подробно описывают шаги выполнения и ожидаемые результаты.

Риск: Длительное время разработки тестов
Решение: Параллельное выполнение ручного тестирования, чтобы не останавливать процесс тестирования на время разработки автоматизированных тестов.

Риск: Проблемы с интеграцией автоматизированных тестов
Решение: Регулярное обновление и проверка тестов, чтобы убедиться в их корректной работе. Внедрение CI/CD процессов для автоматического выполнения тестов при каждом изменении кода.

Риск: Недостаток тестовых данных
Решение: Подготовка резервных данных заранее, чтобы иметь возможность использовать их при необходимости. Автоматизация процесса создания тестовых данных.

Риск: Нестабильность тестируемой версии
Решение: Плотное взаимодействие с разработчиками для быстрой фиксации дефектов. Регулярные обновления и проверки промежуточных версий.

Риск: Проблемы с эмуляторами/симуляторами
Решение: Использование реальных устройств для тестирования в случае возникновения проблем с эмуляторами. Подготовка реальных устройств заранее.

## 14. Тестовая документация на проекте

### Документы
- Тест-план. Содержимое: Цели тестирования, объекты тестирования, методология, типы тестирования, планирование времени, управление рисками, перечень специалистов, тестовая документация, критерии начала и окончания тестирования.
- Чек-листы.Содержимое: Перечень проверок для различных функций приложения.
- Тест-кейсы.Содержимое: Подробное описание шагов для выполнения тестов, ожидаемые результаты.
- Отчеты о ручном тестировании.Содержимое: Результаты выполнения тестов, найденные дефекты.
- Отчеты по автотестам (Allure Reports).Содержимое: Графическое представление результатов автоматизированных тестов, скриншоты

## 15. Критерии начала и окончания тестирования

### Критерии начала тестирования
- Доступны все описания функциональности приложения и сценарии использования, которые объясняют, как приложение должно работать и какие функции оно должно выполнять.
- Подготовлены все необходимые тестовые данные (учетная запись пользователя, тестовые новости и цитаты) . 
- Подготовлена  тестовая среда (включая эмуляторы, симуляторы и реальные устройства).

### Критерии окончания тестирования
- Все запланированные тест-кейсы выполнены, результаты зафиксированы.
- Критические и высокоприоритетные дефекты устранены или получены обходные решения.
- В последнем цикле тестирования не обнаружены критические дефекты.
